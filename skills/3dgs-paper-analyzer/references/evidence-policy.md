# 证据与引用规范

## 1. 证据优先级

优先使用：

1. 官方论文和补充材料；
2. 官方项目主页；
3. 作者官方代码仓库和明确 commit；
4. 官方数据集或 benchmark 文档；
5. 第三方实现；
6. 二手解读。

二手资料可以用于发现线索，但不应在存在一手资料时作为核心技术事实的唯一依据。

## 2. 来源清单

建议维护：

```yaml
paper:
  title:
  version:
  url:
  accessed:
supplement:
  url:
code:
  repository:
  commit_or_ref:
  official: true|false|unclear
project_page:
  url:
other_sources: []
```

## 3. 证据账本

内部使用：

| ID | 结论 | 类型 | 位置 | 直接证据 | 可信度 | 冲突 |
|---|---|---|---|---|---|---|

类型：

- `论文`
- `补充材料`
- `代码`
- `项目主页`
- `数据集文档`
- `独立判断`
- `待核实`

## 4. 证据标签

- `[论文 §3.2]`
- `[论文 Eq. 6]`
- `[论文 Table 1]`
- `[论文 Fig. 4]`
- `[补充材料 Table S3]`
- `[代码 renderer/cuda_rasterizer.cu:L84-L151]`
- `[独立判断]`
- `[待核实]`

不能虚构页码、表号或代码行号。工具无法稳定提供行号时，用“文件路径 + 类/函数名”。

## 5. 数值证据

每个关键数字需要同时说明：

- 数据集；
- 场景或平均方式；
- 指标；
- 评估条件；
- 证据位置。

示例：

> 在 Mip-NeRF 360 的论文定义平均结果上，PSNR 为 25.31 dB，数据来自 Table 1；评估分辨率按作者设置。

速度结果还需要核对：

- FPS 还是毫秒；
- end-to-end 还是 renderer only；
- 分辨率；
- GPU；
- batch size；
- 是否包含数据传输；
- 场景平均方式。

模型大小需要核对：

- checkpoint 还是 bitstream；
- 是否包含 decoder/network；
- 是否包含 optimizer；
- 压缩还是未压缩；
- 单场景还是平均值；
- 是否包含 metadata。

## 6. 可比性等级

- `受控可比`：同一实现或受控复现实验；
- `论文报告、基本可比`：设置表面一致，但来自不同论文；
- `部分可比`：存在分辨率、硬件、版本等差异；
- `不可直接比较`：关键条件不同或缺失。

## 7. 代码证据

强代码映射需要包括：

- 仓库和 ref；
- 文件路径；
- 类、函数或 kernel；
- 配置项；
- 默认值；
- 论文实验是否启用。

README 中写“支持某功能”不能单独证明论文实验实际启用了该功能。

## 8. 独立推断

独立推断应包含：

1. 观察到的事实；
2. 推理过程；
3. 推断结论；
4. 不确定性。

不能写：

> 作者故意选择了有利场景。

除非存在直接证据。

可以写：

> 论文选用的场景在类别 X 中占比更高，且未说明选择规则，因此该分类结论可能存在场景选择偏差。

## 9. 信息冲突

论文、补充材料、项目主页和代码冲突时：

1. 分别保留；
2. 标注版本或 commit；
3. 解释实际影响；
4. 不静默选择其中一个。

## 10. 可信度

- `高`：一手来源直接支持且内部一致；
- `中`：有支持，但缺少一个重要条件；
- `低`：间接证据、推断或存在冲突；
- `未知`：证据不足。
