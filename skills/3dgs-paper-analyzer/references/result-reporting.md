# 结果汇报规范

本文件解决“分析了方法，但没有把结果汇报清楚”的问题。

任何模式都不能只写“效果更好”“性能提升明显”。必须回答：

- 在什么数据集上；
- 使用什么指标；
- 相比哪个 baseline；
- 提升或下降多少；
- 是否公平可比；
- 是否在多数场景成立；
- 付出了什么代价；
- 哪些结论被实验支持；
- 哪些结论尚未得到支持。

## 1. 开头汇报摘要

报告开头必须有：

| 项目 | 结论 |
|---|---|
| 任务 | |
| 方法核心 | |
| 最重要定量结果 | |
| 效率/存储结果 | |
| 最大优势 | |
| 主要代价 | |
| 代码状态 | |
| 总体判断 | |

如果论文未提供某项，写“论文未报告”，不能删掉该行。

## 2. 主要结果总表

建议格式：

| 数据集/场景 | 指标 | 本文 | 可比 baseline | 差值 | 趋势 | 可比性 | 证据 |
|---|---:|---:|---:|---:|---|---|---|

规则：

- 不同指标分行；
- 不同数据集分行；
- 不可比时不计算差值；
- 最多优先列出 5—12 个最重要结果；
- 全量数据过多时放附表，并在正文概括。

## 4. 逐场景一致性

不能只看平均值。需要判断：

- 多数场景提升还是少数场景拉高均值；
- 是否存在明显退化场景；
- 室内/室外、有界/无界是否表现一致；
- 论文结论是否跨数据集稳定；
- 是否只展示有利场景。

可使用：

| 数据集 | 提升场景数 | 退化场景数 | 最大提升 | 最大退化 | 是否稳定 |
|---|---:|---:|---:|---:|---|

找不到逐场景数据时，明确写“论文仅报告平均值，无法判断场景一致性”。

## 5. 消融实验汇报

| 组件 | 去除后变化 | 说明了什么 | 是否存在混杂因素 | 证据 |
|---|---|---|---|---|

结果解读不能只复述数字，需要说明：

- 哪个组件贡献最大；
- 是否有组件几乎无贡献；
- 是否存在组件组合效应；
- 消融是否同时改变多个因素；
- 是否覆盖主要数据集。

## 6. 定性结果

需要说明：

- 图中最明显的改善；
- 失败区域；
- 是否与定量结果一致；
- 是否只展示少数案例；
- 是否缺少 challenging case。

不要把视觉上更好直接等同于整体指标更好。

## 7. 收益—代价分析

必须回答：

| 收益 | 代价 |
|---|---|
| 质量提升 | 训练时间 |
| 几何改善 | 模型大小 |
| 渲染加速 | 质量损失 |
| 存储压缩 | 解码复杂度 |
| 泛化能力 | 外部预训练依赖 |

## 8. “结果证明了什么”

用 2—6 条陈述，只写被证据充分支持的结论。

示例：

- 在论文使用的三个数据集和相同分辨率设置下，模块 A 能稳定降低模型大小。
- 消融实验支持 loss B 对薄结构重建有独立贡献。

## 9. “结果尚未证明什么”

用 2—6 条陈述指出边界。

示例：

- 未提供跨硬件速度测试，因此不能证明移动端实时性。
- 只选择部分场景，无法确认有界/无界分类结论对完整数据集成立。
- 未提供真实 bitstream，因此无法验证实际编码开销。

## 10. 结果汇报总结

最终必须给出一段可直接用于组会或汇报的总结，至少包含：

1. 最重要结果；
2. 稳定性；
3. 代价；
4. 最有说服力的实验；
5. 最薄弱的证据；
6. 总体判断。

推荐格式：

- **最重要结果：**
- **提升是否稳定：**
- **主要代价：**
- **最有说服力的实验：**
- **证据最薄弱的主张：**
- **结果层面的总体判断：**
